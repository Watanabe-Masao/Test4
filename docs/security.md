# セキュリティ

本システムのセキュリティ設計とデータ保護の方針について説明します。

## データプライバシー

### ブラウザ内完結アーキテクチャ

本システムは**完全にブラウザ内で動作する SPA（Single Page Application）**です。

- インポートされたデータは**外部サーバーに送信されません**
- 全ての計算処理（粗利計算、要因分解、DuckDB SQL 分析）はブラウザ内で実行されます
- ネットワーク通信は静的アセット（HTML/JS/CSS/WASM）の配信時のみ発生します
- API サーバーへのデータ送信は一切ありません

### データの保存先

| 保存先 | 用途 | ライフサイクル |
|---|---|---|
| IndexedDB | インポートデータの永続化 | ユーザーが明示的に削除するか、ブラウザのサイトデータクリアまで保持 |
| DuckDB-WASM（メモリ） | SQL 分析用テーブル | ページリロードで消失（IndexedDB から再ロード） |
| localStorage | テーマ設定等の軽量データ | サイトデータクリアまで保持 |

## IndexedDB の分離

### オリジンベースのアクセス制御

IndexedDB はブラウザの**同一オリジンポリシー**に基づくアクセス制御を適用します。

- 同じオリジン（プロトコル + ドメイン + ポート）のページのみがデータにアクセス可能
- 異なるオリジンのサイトからは本システムの IndexedDB にアクセスできません
- 各ユーザーのブラウザ環境ごとにデータが完全に分離されます

### 注意事項

- 同じ端末の同じブラウザプロファイルを共有するユーザー間ではデータが見える可能性があります
- 共用端末で使用する場合は、使用後にブラウザのサイトデータをクリアすることを推奨します

## 入力検証

### FileImportService のバリデーション

`FileImportService`（`application/usecases/import/FileImportService.ts`）は
ファイルインポート時に以下の検証を行います:

1. **ファイル種別判定**: ファイル名パターン・カラム構造に基づく種別の自動判定
2. **スキーマ検証**: 必須カラムの存在チェック、データ型の妥当性検証
3. **データ整合性チェック**: データソース間（classifiedSales vs categoryTimeSales）の
   売上合計乖離が 1% 以上の場合に警告
4. **異常値検知**: 数値カラムへの文字列混入、NaN、負値の検出

### 不正なファイルの拒否

- サポート外のファイル形式はインポートを拒否します
- 破損したファイルや解析不能なファイルはエラーメッセージとともに拒否します
- パスワード保護された Excel ファイルは読み込みできません

## XSS 対策

### React の自動エスケープ

本システムは React 19 を使用しており、JSX の標準的な自動エスケープ機構により
ユーザー入力が HTML として解釈されることを防ぎます。

- `dangerouslySetInnerHTML` は使用していません
- ファイル名・店舗名等のユーザー由来文字列は React の通常のレンダリングパスで表示されます

### DuckDB SQL インジェクション対策

DuckDB へのクエリ生成において、以下の対策を実施しています:

#### storeIdFilter でのクォートエスケープ

```typescript
// queryRunner.ts
export function storeIdFilter(storeIds: readonly string[] | undefined): string | null {
  if (!storeIds || storeIds.length === 0) return null
  // シングルクォートを '' にエスケープして SQL インジェクションを防止
  const quoted = storeIds.map((id) => `'${id.replace(/'/g, "''")}'`).join(', ')
  return `store_id IN (${quoted})`
}
```

#### クエリパラメータの扱い

- ユーザー入力（店舗 ID 等）が SQL に埋め込まれる箇所では、必ずエスケープ処理を適用
- 数値パラメータ（year, month, day）は TypeScript の型システムで `number` 型に制約
- `buildWhereClause` は null 安全な条件結合を行い、不正な条件の混入を防止

### リスク軽減要因

- DuckDB-WASM はブラウザ内のインメモリ DB であり、外部の永続データベースへの接続はない
- SQL インジェクションが成功した場合でも、影響範囲はブラウザ内のセッションデータに限定される
- ユーザー自身がインポートしたデータのみがテーブルに存在する

## 依存ライブラリのセキュリティ

### npm audit

定期的に `npm audit` を実行し、依存ライブラリの既知の脆弱性を確認します。

```bash
cd app
npm audit
```

### 主要な依存ライブラリ

| ライブラリ | バージョン | 用途 |
|---|---|---|
| `react` | ^19.2.0 | UI フレームワーク |
| `@duckdb/duckdb-wasm` | ^1.33.1 | ブラウザ内 SQL エンジン |
| `styled-components` | ^6.3.9 | CSS-in-JS |
| `xlsx` | ^0.18.5 | Excel ファイル読み込み |
| `zod` | ^4.3.6 | スキーマバリデーション |
| `zustand` | ^5.0.11 | 状態管理 |
| `recharts` | ^3.7.0 | チャート描画 |

### 推奨プラクティス

1. **定期的な依存更新**: `npm audit` を CI パイプラインに組み込み、脆弱性を早期検知
2. **lock ファイルの管理**: `package-lock.json` をバージョン管理に含め、再現可能なビルドを保証
3. **最小権限の原則**: 不要な依存ライブラリを追加しない
4. **WASM バイナリの検証**: DuckDB-WASM の WASM バイナリは npm パッケージから取得
   （CDN 等の外部ソースには依存しない）
