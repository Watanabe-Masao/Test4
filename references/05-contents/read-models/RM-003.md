# RM-003 — 売上・販売点数 ReadModel

## 1. 概要

`buildSalesFactReadModel(rows)` は **pure 同期** builder で、`SalesFactRow[]`（DuckDB query 出力）を `SalesFactReadModel`（grand 合計 + dept / store / daily / hourly の 5 view）に集約する正本（`sales-definition.md` 準拠）。

## 2. Input Contract

- `rows: readonly SalesFactRow[]` — DuckDB SQL 結果を Zod parse 済み（`SalesFactQueryInput.rowSchema.parse`）
- 各 row: `date` / `storeId` / `deptCode` / `lineCode` / `klassCode` / `hour` / `salesAmount` / `salesQty`

## 3. Output Contract

`SalesFactReadModel`:

- `grandTotals`: `totalSalesAmount` / `totalSalesQty`（不変条件: rows.sum と一致）
- `byStore` / `byDept` / `daily` / `hourly` の 4 view（同じ raw を group-by で射影）
- `meta.sourceDate`（domain 整合）

派生 helper（同 module export）:
- `toStoreSalesRows(model)` — store-level rows
- `toDailySalesRows(model)` — daily rows
- `toHourlySalesRows(model)` — hour x day 24×N matrix
- `toDeptSalesRows(model)` — dept-level rows

## 4. Upstream Dependencies

- `salesFactHandler` (`QueryHandler<SalesFactInput, SalesFactOutput>`) — DuckDB execution
- `comparisonQueryScope.ts` — current / comparison の 2 並列パターン

## 5. Consumers

- `useWidgetDataOrchestrator` で配布
- `IntegratedSalesChart` / `PerformanceIndexChart` / `CategoryHierarchyExplorer` 等
- `useStoreDailyBundle` / `useCategoryDailyBundle` の上流入力

## 6. Fallback / Readiness

- 入力 0 件 → `grandTotals = { totalSalesAmount: 0, totalSalesQty: 0 }` を返す（pure builder のため fail しない）
- query 自体が reject の場合は handler 側で `status: 'error'`、本 builder には到達しない

## 7. Invariants

- INV-SF-01: `grandTotals.totalSalesAmount` = sum(rows.salesAmount)
- INV-SF-02: `grandTotals.totalSalesQty` = sum(rows.salesQty)
- INV-SF-03: 4 view の合計値は grand と一致（group-by の同値性）

詳細: `sales-definition.md` §「販売点数 vs 来店客数の意味分離」

## 8. Co-Change Impact

- `SalesFactRow` schema 変更 → builder の field mapping 破壊
- `SalesFactReadModel` の view 構造変更 → 全 consumer の field access 破壊
- `to*Rows` helper の signature 変更 → chart input builder の破壊（`chartInputBuilderGuard` 監視）
