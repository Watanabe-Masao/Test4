# RM-001 — 仕入原価 ReadModel

## 1. 概要

`readPurchaseCost(input)` は 3 系統の仕入原価ファクト（`typeIn` / `typeOut` / `costInclusion`）を `QueryPort` 経由で並列取得し、単一 `PurchaseCostReadModel` に統合する **async 正本 builder**（`purchase-cost-definition.md` の 3 独立正本統合契約に準拠）。

## 2. Input Contract

- `PurchaseCostInput extends BaseQueryInput` — `dateFrom` / `dateTo` / `selectedStoreIds`
- 並列実行する 3 内部クエリ: `typeIn` / `typeOut` / `costInclusion`
- 全クエリは `comparisonScope` を共有（current / comparison の 2 並列形式は呼び出し側で対応）

## 3. Output Contract

`PurchaseCostReadModel` は以下を保証:

- `dailyByStore` / `dailyByDept` / `dailyByCategoryLeaf` の 3 view（同じ raw fact から派生）
- `totals.typeInAmount` + `totals.typeOutAmount` + `totals.costInclusionAmount` の grand 合計が 3 ファクトの sum と一致（不変）
- `meta.sourceDate` を保持（domain 整合性のため）

## 4. Upstream Dependencies

- `QueryPort.execute` (3 並列): `purchase_typein_*` / `purchase_typeout_*` / `cost_inclusion_*` SQL
- `infrastructure/duckdb/schemas` の 3 テーブル契約

## 5. Consumers

- `useWidgetDataOrchestrator` 経由で全 widget に配布
- `CostDetail` ページの `useCostDetailData` 経由（features/cost-detail）
- `purchaseCostHandler` (`QueryHandler<PurchaseCostInput, PurchaseCostOutput>`) として `createPairedHandler` でも消費可能

## 6. Fallback / Readiness

- 3 並列クエリのいずれかが reject → `Promise.all` が reject、上位 readiness で `status: 'error'` に分類
- 0 件結果（空配列）は ready 扱い、`totals.* = 0` を返す

## 7. Invariants

- INV-PCD-01: `totals.typeInAmount` = sum(`typeIn`)
- INV-PCD-02: `totals.typeOutAmount` = sum(`typeOut`)
- INV-PCD-03: `totals.costInclusionAmount` = sum(`costInclusion`)
- INV-PCD-04: 3 ファクトの `dateRange` / `selectedStoreIds` が完全一致（独立正本の同期保証）

詳細: `purchase-cost-definition.md` §「3 独立正本の統合契約」
