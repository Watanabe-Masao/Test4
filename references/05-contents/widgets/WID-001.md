# WID-001 — 店別予算達成状況

## 1. 概要

`allStoreResults` が非空の場合に `ConditionSummaryEnhanced` を full ctx passthrough でレンダリングする、予算進捗ハブの単一 widget（registryKpiWidgets.tsx:19-25）。

## 2. Data Acquisition Path

- registry 定義行では取得ロジックを持たず、full ctx を `ConditionSummaryEnhanced` に委譲（`acquisitionPath: ctx-direct`）
- 実際のデータ取得は子 component 側の `ConditionSummaryEnhanced.vm` の pure builder 群（`buildCardSummaries` / `buildBudgetHeader` / `buildYoYCards` / `buildUnifiedCards`）が `ctx` から抽出（本 spec の frontmatter `consumedCtxFields` は registry 行からの直接参照のみを列挙する方針に従い、子側の touch は本節で言語化）

## 3. Context Fields Consumed — 解説

- registry 行から直接参照されるのは `allStoreResults` のみ（isVisible predicate）
- render 関数は `ctx` を untyped に forward するため、**実参照は子 component の実装に散在**する。inquiry/01 §b の ambiguity「子 component が触る ctx フィールドは本台帳で未追跡」に該当
- Dashboard-local `WidgetContext` の required フィールドほぼ全てが子側で潜在的に参照可能

## 4. Composition

- 直接の子: `ConditionSummaryEnhanced`（`registryKpiWidgets.tsx:2` で import）
- `ConditionSummaryEnhanced.tsx` 自身が更に以下を import して合成:
  - `ScrollableCardRow`（`ConditionScrollableCardRow`）
  - `ConditionSummaryBudgetDrill`
  - `ConditionSettingsPanelWidget`
  - `YoYDrillOverlay`
- pure builder: `ConditionSummaryEnhanced.vm` が `buildCardSummaries` / `buildBudgetHeader` / `buildYoYCards` / `buildUnifiedCards` / `computeTrend` / `computeRateTrend` を export
- 外部 import: `extractPrevYearCustomerCount`（`@/features/comparison`）、`formatPercent`（`@/domain/formatting`）

## 5. Visibility

- 条件: `({ allStoreResults }) => allStoreResults.size > 0`（`registryKpiWidgets.tsx:23`）
- `allStoreResults: ReadonlyMap<string, StoreResult>`。空 Map では非可視、1 件以上で可視
- Dashboard-local `WidgetContext` では `allStoreResults` は required（`Dashboard/widgets/types.ts:75`）。`size > 0` の防御的チェックは、runtime で空 Map が到達しうる経路の存在を示唆

## 6. Invariants

- 本 widget 自身は数学的不変条件を持たない（registry 層は view marshalling のみ）
- 子 builder 群（`ConditionSummaryEnhanced.vm`）が扱う予算達成率・前年比計算が上位の invariant（`references/03-guides/invariant-catalog.md` の予算・YoY 系）に従属

## 7. Co-Change Impact

以下が変わると本 widget が壊れうる:

- `WidgetContext` の `allStoreResults` field 型変更 → isVisible predicate 破綻
- `ConditionSummaryEnhanced` の props shape 変更（現状 `{ ctx: WidgetContext }`）→ render 行の型エラー
- `ConditionSummaryEnhanced.vm` の builder signature 変更 → 子 component 側の連鎖破壊（本 widget 直接 ref ではないが、子経由で影響）
- Dashboard-local `WidgetDef` interface（`Dashboard/widgets/types.ts:101`）変更

## 8. Guard / Rule References

- `responsibilityTagGuard.test.ts`（責務タグ R:component 期待）
- `presentationIsolationGuard.test.ts`（presentation 層 isolation）
- `sizeGuard.test.ts`（ConditionSummaryEnhanced ファミリのサイズ監視）
- 本 widget 固有の guard は無し（5 層構造に沿う単純 widget）

## 9. Pipeline Concerns / Upstream Requests

> 事実のみを記録する。widget 側 workaround / 「〜すべき」は書かない（`references/05-contents/README.md` §禁止パターン参照）

- **子 component への full ctx passthrough による静的解析不能**: render 行は `<ConditionSummaryEnhanced ctx={ctx} />` として `WidgetContext` 全体を untyped に forward。frontmatter `consumedCtxFields` の生成（Phase 6）では子内部の AST 走査を本 widget の依存として統合する必要がある。該当責務層: presentation（本 widget の再設計）または application（子に渡す ctx を絞り込む adapter の導入）
- **子側の `useSettingsStore` 直接参照**: `ConditionSummaryEnhanced.tsx:24` が `useSettingsStore` を import。widget レイヤーから store を直接触る形で、本 widget 自身は ctx のみを受けているにもかかわらず子が store と ctx の両方を読むパターン。該当責務層: application（store 読取を hook に束ねる）
- **`allStoreResults` の空 Map 経路**: Dashboard-local `WidgetContext` で required とされる `allStoreResults` に対し、registry 行で `size > 0` の防御的チェックが必要な経路が存在する。required 宣言と runtime 実態の乖離を示唆。該当責務層: application（`useUnifiedWidgetContext` 側の `allStoreResults` 構築経路）
- **外部 feature import の散在**: `ConditionSummaryEnhanced.tsx:20` が `@/features/comparison` から `extractPrevYearCustomerCount` を直接 import。presentation 配下の widget が features/ の API を直接消費する経路がある。該当責務層: application（`readModels` または `comparisonScope` への移設の妥当性検証）
