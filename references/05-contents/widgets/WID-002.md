# WID-002 — 日別売上チャート

## 1. 概要

17 個の ctx field を個別 props として `IntegratedSalesChart` に渡し、かつ `widgetCtx={ctx}` で full ctx も重ねて注入する、日次時系列の総合表示 widget（registryChartWidgets.tsx:32-58）。

## 2. Data Acquisition Path

- registry 行で `ctx` を分解し、17 field を名前付き props として子に渡す（`acquisitionPath: ctx-direct`）
- `ctx.result.daily` / `ctx.result.budgetDaily` / `ctx.result.discountEntries` / `ctx.result.grossSales` の 4 field が `StoreResult`（`ctx.result`）を解体して個別に forward
- 天気 (`weatherDaily` / `prevYearWeatherDaily` / `weatherPersist`) は ETRN fallback 系の 3 field を独立に渡す
- 同時に `widgetCtx={ctx}` として full ctx を子に duplicate 注入（registryChartWidgets.tsx:56）
- 取得経路として `queryExecutor` / `currentDateRange` / `selectedStoreIds` / `prevYearScope` の組も子に直接渡す（子内で DuckDB query を発行する可能性を示唆）

## 3. Context Fields Consumed — 解説

- 17 field 参照は本 project の widget 内で最大級（inquiry/01 registryChartWidgets.tsx:32-58 で実測）
- `result` を分解して `result.daily` / `result.budgetDaily` / `result.discountEntries` / `result.grossSales` を個別展開している。StoreResult の内部構造が本 registry 行で surface されている
- `prevYear.hasPrevYear ? prevYear.daily : undefined` の条件転送で、optional 意味を子に引き継ぐ
- `comparisonScope?.dowOffset ?? 0` で null-safe access を registry 行で実施

## 4. Composition

- 直接の子: `IntegratedSalesChart`
- registry 行が 17 field の props を展開。子 component は更に `widgetCtx` full ctx も受けており、**props 経由と ctx 経由の二重データ注入**の形
- 子内部の実装は本 spec では未追跡（inquiry/01 §b ambiguity）

## 5. Visibility

- `isVisible` 未指定（registryChartWidgets.tsx に `isVisible` プロパティなし）→ 常時可視
- 可視条件は子 component 内で handle される前提（本 registry 行では制御しない）

## 6. Invariants

- 本 widget 自身は数学的不変条件を持たない（registry 層は props marshalling のみ）
- `result.daily` / `result.budgetDaily` / `prevYear.daily` の配列長整合は StoreResult 正本側の責務（`references/03-guides/invariant-catalog.md` の日次系列不変条件に従属）
- `comparisonScope.dowOffset` が 0 の fallback（未定義時）は本 widget 行で実施

## 9. Pipeline Concerns / Upstream Requests

> 事実のみを記録する。widget 側 workaround / 「〜すべき」は書かない

- **17 field の個別 props 展開による契約の広さ**: registry 行が `StoreResult` を解体（`result.daily` / `result.budgetDaily` / `result.discountEntries` / `result.grossSales`）+ `prevYear` 解体 + 天気 3 field + 取得経路 4 field（queryExecutor / currentDateRange / selectedStoreIds / prevYearScope）+ comparisonScope 1 field + weatherPersist + ctx 全体、を一度に forward。widget と子の契約面積が極大。該当責務層: application（`IntegratedSalesChart` への入力を束ねる adapter / plan hook の妥当性検証）
- **props 経由 + full ctx 経由の二重注入**: `widgetCtx={ctx}` により、17 field 展開とは別に full ctx も子に渡される。17 props の一部は子内部で `widgetCtx.X` としても再参照可能な構造。どちらの経路を正とするか unresolve。該当責務層: application（子 component の入力契約整理）
- **子内部の DuckDB query 発行経路**: `queryExecutor` / `currentDateRange` / `selectedStoreIds` / `prevYearScope` が子に直接渡されている。registry 層の widget が query 発行 surface を子にそのまま forward している形。Screen Plan を経由する標準経路（CLAUDE.md H1 原則）とは別経路。該当責務層: application（`useQueryWithHandler` 経由への移行可否検証、`references/01-principles/runtime-data-path.md` 照合）
- **ETRN fallback 天気の registry 層 surface**: `weatherPersist` / `weatherDaily` / `prevYearWeatherDaily` が registry 行で個別に forward。天気データの fallback 契約が widget 登録層まで漏れ出している。該当責務層: application（weather lane 化、`references/01-principles/data-flow.md` の lane 原則照合）
- **`comparisonScope?.dowOffset ?? 0` の registry 行での null-safe access**: `WidgetContext` の `comparisonScope: ComparisonScope | null` が required だが null 許容型であり、registry 行で 0 fallback を決定している。`dowOffset` の意味論（「比較対象なし → 0」）が presentation 最表層で固定されている。該当責務層: domain または application（`ComparisonScope` の default 値責務の所在確認）
- **`inquiry/01 §f.1` 対応**: 本 widget は `WIDGETS_CHART` (`WidgetDef` 型 A = Dashboard-local) だが、`linkTo: { view: 'daily' }` で Daily ページ（`DAILY_WIDGETS` = `WidgetDef` 型 B = Unified）への遷移 hint を持つ。型 A / B を跨ぐリンクが存在する事実
