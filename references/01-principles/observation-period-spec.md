# 観測期間仕様

## 1. 目的

予測、前年比較、曜日差分析、予算進捗などの時間系ロジックで使う基準時点を統一し、
ロジックごとの個別判断や月跨ぎ補完を防ぐ。

本仕様の目的は、**営業日そのものを厳密定義することではなく、分析計算で使う観測済み範囲を一意に定義すること**にある。

## 2. 基本方針

観測期間は **最終売上計上日ベース** で定義する。

### 原則

1. 観測期間は対象月内で閉じる
2. 月跨ぎ補完は行わない
3. 不足は補完せず、状態として返す
4. 各ロジックで独自に観測期間を解釈しない
5. authoritative な指標と exploratory な指標で採用ルールを分ける

## 3. 用語定義

| 用語 | 定義 |
|---|---|
| **対象月** | 分析対象となる年月 |
| **最終売上計上日** (`lastRecordedSalesDay`) | 対象月内で、売上計上が確認できる最も遅い日 |
| **観測期間** | 対象月の月初から `lastRecordedSalesDay` までの範囲 |
| **経過日数** (`elapsedDays`) | 月初から `lastRecordedSalesDay` までの日数 |
| **残日数** (`remainingDays`) | `daysInMonth - lastRecordedSalesDay` |
| **売上計上日数** (`salesDays`) | 観測期間内で売上計上実績が存在する日数 |
| **比較可能日数** | 前年同月または比較対象月において、同一月内で対応付け可能な日数 |
| **coverageRatio** | 必要比較日数に対して、実際に取得できた比較可能日数の割合 |

## 4. 観測期間の決定ルール

### 当年側

- 観測期間は必ず **対象月の月初〜最終売上計上日** とする
- `lastRecordedSalesDay` が存在しない場合、その月の観測期間は未確定とする

### 前年側・比較側

- 比較対象の取得も **対象月内で閉じる**
- 前月・翌月からの補完はしない
- 比較可能日数が不足する場合は、その不足を状態として返す

## 5. 禁止事項

- 前年比較日数不足を前月・翌月で補う
- ロジックごとに観測終了日を別定義にする
- 不足を無言でカレンダー日数へ変換する
- `partial` や `invalid` を 0 や正常値として扱う
- explanation 上で不足を明示せずに正式値として説明する

## 6. 結果状態

観測期間や比較可能性を伴う計算は、少なくとも次の状態を返せるようにする。

### status

| status | 定義 |
|---|---|
| `ok` | 観測期間・比較条件ともに十分 |
| `partial` | 値は出せるが、比較可能日数不足や観測不足がある |
| `invalid` | 定義上、比較や予測の根拠が不足していて正式値として扱えない |
| `undefined` | 入力不足により計算不能 |

### warnings の例

- `last_recorded_sales_day_missing`
- `prior_year_period_insufficient`
- `month_boundary_truncated`
- `insufficient_comparable_days`
- `observation_window_incomplete`

## 7. authoritative / exploratory の採用ルール

### authoritative 指標

- `status = ok` のみ原則採用
- `partial` は原則採用しない
- 例外採用する場合は、Metric Definition Registry に明示的な fallback rule を持つ

### exploratory 指標

- `partial` でも表示可能
- warning を必須表示
- explanation では「参考値」であることを明示する

## 8. 予測系ロジックへの適用

### 予測の観測済み範囲

- 月初〜`lastRecordedSalesDay`

### 平均値の分母

- 仕様で定めた基準に従う
- 例: `salesDays` を使うなら、予測側も同概念と整合する基準を使う

### 注意

**「営業日平均 × 残カレンダー日数」のような、分母と乗数の基準が異なる計算は採用しない。**
基準を変える場合は、別仕様として明示する。

## 9. 前年比較・曜日比較への適用

### 前年比較

- 比較対象月は同月内で閉じる
- 月跨ぎ補完をしない
- 不足時は `partial` または `invalid`

### 曜日比較

- 曜日構成差と coverage は別概念として扱う
- coverage が十分でも曜日構成が大きく異なる場合は warning 対象とする

## 10. 最低必要日数

比較や予測には最低必要日数を設ける（暫定ルール）。

| 日数 | status |
|---|---|
| 0〜4日 | `invalid` |
| 5〜9日 | `partial` |
| 10日以上 | `ok` |

この閾値は用途ごとに調整可能だが、ロジックごとの独自定義は禁止し、共通設定から参照する。

## 11. 実装責務

| Engine | 責務 |
|---|---|
| **TypeScript** | 観測期間の意味づけ、status / warning の決定、authoritative / exploratory の採用判断 |
| **DuckDB / SQL** | 観測期間に必要な集約素材の生成、日別・月別・比較用の中間集計 |
| **Rust / WASM** | 本仕様の結果を受けた純粋数理計算のみ。観測期間そのものの解釈は行わない |

## 12. 今後の拡張余地

将来的に実営業日マスタを導入する場合でも、本仕様の基本構造は維持する。
その場合は `lastRecordedSalesDay` ベースの観測期間と実営業日ベースの補助情報を併用し、
観測期間の正本は引き続き一元管理する。

## 13. 採用判断

本仕様では、分析計算の時間軸は **「営業日」ではなく「最終売上計上日ベースの観測期間」** を正本とする。
これにより、月跨ぎ判定や補完判断を各ロジックへ分散させず、観測済み範囲の定義を一元化する。
