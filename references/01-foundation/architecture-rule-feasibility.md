# Architecture Rule System — Feasibility Assessment

## 思想とルールの関係

> **思想は普遍である。ルールは文脈依存である。**

ルールは思想を実現するための手段であり、目的ではない。
ルールが現在の文脈に最も適切だと判断してルール化するが、それは普遍的なものではない。
仮にルールが普遍的だとしたら、それは成長がないことを意味する。

| | 性質 | 変化 | 例 |
|---|---|---|---|
| **思想** | 普遍 | 変わらない | 「気をつけるで終わらせない」「現実を正しく把握する」 |
| **ルール** | 文脈依存 | 変わるべき | useMemo ≤ 7、baseline: 48 |

ルールが思想に奉仕している間は健全。
ルールが思想の実現を阻害し始めたら、ルールを変えるべきである。

**監視すべきは「ルールが守られているか」ではなく
「ルールが思想に沿って成長を阻害していないか」である。**

maturity lifecycle（experimental → stable → deprecated）と例外圧検出は、
この監視を機械的に行うための仕組みである。

### ルールを疑い、捨て、置き換える 5 つの原則

> **1. 反証可能性**
>
> 良いルールとは、守られるルールではなく、間違っていた場合にそれが分かるルールである。
> 「いつこのルールが不要になるか」を持たないルールは未完成である。

> **2. ルールは負債でもある**
>
> ルールは資産であると同時に負債である。
> 守らせるコスト、学習コスト、例外処理コスト、誤誘導コストを持つ。
> ルール新設は制約追加ではなく運用負債の発行とみなす。

> **3. 例外は観測である**
>
> 例外はルール逸脱ではなく、ルール理解のための観測点である。
> 例外の存在は即ち悪ではない。
> 反復される例外は、実装の怠慢ではなくルールの粗さを先に疑う。

> **4. 探索の権利と出口の対称性**
>
> 探索は制約違反ではなく、制約以前の観測である。
> 新しいパターンは観測期間を経て初めて gate になる。
> 様子見のルールは、採用条件だけでなく撤回条件を持たなければならない。
> 観測のために導入したルールが、観測終了後も存在し続けるなら、それは統治ではなく惰性である。

> **5. 撤退の美学**
>
> 良い統治は、増やすことより減らすことに現れる。
> 成熟度はルール数ではなく、不要なルールを安全に削除できる能力で測る。
> 新しいルールを作る能力より、不要なルールを廃止する能力の方が成熟度を示す。

### 6. 時間軸を持たない状態は滞留である

> 時間軸を持たない例外・互換・移行状態は、制度ではなく滞留である。
> 撤回条件より先に、撤回を忘れない仕組みを入れること。
> ルールに時計を持たせ、「いつ再点検するか」を制度化する。

reviewPolicy（owner / lastReviewedAt / reviewCadenceDays）は、
「正しさを疑い続ける制度」の実装である。

### 7. ルールの価値は害の削減で測る

> 良いルールとは、守られているルールではなく、害を減らしているルールである。
> ルールの成熟は、定義の精密さやレビュー頻度ではなく、
> 誤修正・再発・例外依存を実際に減らせているかで判断する。

protectedHarm（このルールが防いでいる具体的な害）と
efficacy 指標（高例外圧・renewal 合計・dormant ルール）は、
「制度が整っているか」ではなく「制度が成果に結びついているか」を測る仕組みである。

### 整然さではなく現実適合

整然としていることは価値ではない。現実に適合していることが価値である。
ルールはコードを美しくするためではなく、
誤判断・誤修正・誤集約を減らすために存在する。

## 残存する構造的抜け穴

本システムは「法・監査・台帳」として成熟しているが、
「感覚・学習・政治・代謝」にはまだ弱点がある。

### 1. 「健康」と「目的達成」のズレ

governance health（review overdue なし、baseline 内）は測れるが、
プロダクト価値や開発成果との直結はまだ弱い。
→ 将来: revert 率、再発率、incident-rule 対応などの Outcome Health 層

### 2. 免疫系が新規性を攻撃する危険

rule system は「壊す変更」だけでなく「見慣れない変更」も攻撃しうる。
experimental + warn で緩和しているが、探索的な突然変異を潰すリスクは残る。
→ 将来: quarantine mode（observation-only の隔離期間）

### 3. review の儀式化

lastReviewedAt は更新されるが中身を見ていない可能性。
→ 将来: review を日付ではなく差分付き判断履歴（decision + rationale + evidence）に

### 4. 例外の「身分化」

active-debt が居座り、permanent が神聖化される。
renewalCount はあるが、実際に消す圧力がまだ弱い。
→ 将来: 退出予算（N 回延長で fail、M 日で escalation）

### 5. 法律はあるが政治がない

rule は優先順位の対立（correctness vs migration-speed vs local-simplicity）を内包するが、
その調停制度がない。
→ 将来: tradeoffPriority を rule に持たせる

### 6. 記憶はあるが学習が弱い

createdAt / reviewPolicy / health は記録に強いが、
「前の失敗がどう rule に反映されたか」「rule が不要だったケース」の学習機構がない。
→ 将来: incident-to-rule lineage（bornFromIncident / reinforcedBy / invalidatedBy）

### 7. 中央統治に寄りすぎる

局所的な痛みや工夫が中央に吸い上がるまで遅い。
→ 将来: local adaptation channel（hotspot 自動エスカレーション）

### 8. all-green の自己欺瞞

指標は全部 OK だが現場はつらい、という状態を検出できない。
→ 将来: green-but-painful audit（guard を通ったが苦しかった変更の収集）

### 対処の優先順位

上記 8 つは全て正しいが、全てを同時に入れると制度の複雑性が爆発する。
対処順は以下の通り:

1. **Evidence Governance**（protectedHarm + efficacy KPI）← **実装済み**
2. review の差分化（decision log）
3. 例外の退出予算
4. Outcome Health 層
5. tradeoffPriority
6. quarantine mode
7. incident-to-rule lineage
8. local adaptation / green-but-painful audit

**1 人 + AI 開発において、PR レビューアが不在の状態で
アーキテクチャの設計品質をどう維持するか。**

| 課題 | 従来の解決手段 | 本プロジェクトでの問題 |
|---|---|---|
| 設計判断の伝達 | PR レビュー | レビューアがいない |
| 設計原則の強制 | チームの暗黙知 | AI は毎セッション初対面 |
| 品質の段階的改善 | スプリント回顧 | user 1 人では改善が止まる |
| 判断の一貫性 | チーム文化 | AI に文化は蓄積されない |

Architecture Rule は上記 4 つの課題を **構造化データ + CI テスト** で解決する仕組みである。

## 1. システムの位置づけ

Architecture Rule は **1 人 + AI 運用で成熟したアーキテクチャを維持する** ための
実行可能なアーキテクチャ仕様である。

通常のチーム開発では PR レビューが担う「設計判断の伝達と強制」を、
構造化データ（84 ルール）と CI テスト（39 ガード）で代替する。

### 前提条件（このシステムが成立する条件）

1. **開発者が 1 人 + AI** — PR レビューアがいない
2. **AI は毎セッション「初対面」** — 暗黙知が通用しない
3. **アーキテクチャが成熟済み** — 設計原則が 8 カテゴリ・50+ タグで確立されている
4. **ガードテストの文化がある** — 39 ファイル / 290+ テストの実績

上記のいずれかが欠けると、このシステムのコスト対効果は大きく下がる。

## 2. メリット

### M1: AI セッション間の知識断絶を解決

| 比較 | 学習コスト |
|---|---|
| CLAUDE.md（700行散文） | AI が解釈して構造化する必要がある |
| references/（20+ ファイル） | どのファイルを読むべきか判断が必要 |
| **architectureRules.ts** | **構造化データとして即座にパース可能** |

AI にとって散文より構造化データのほうが圧倒的にパース効率が高い。
1 ファイルで 84 ルールの what / why / correctPattern / migrationPath /
decisionCriteria / relationships が手に入る。

**定量的効果**: セッション開始時のコンテキスト構築時間を短縮。
AI が「なぜこのルールがあるのか」を毎回質問する必要がなくなる。

### M2: 「気をつける」を「通らない」に変換

設計判断が暗黙知ではなく CI で強制される。
1 人開発で最も危険な「自分ルールの忘却」を機械的に防止する。

**具体例**: `AR-A1-PRES-INFRA` — presentation/ から infrastructure/ への
value import は CI で即座にブロックされる。「うっかり」が存在しない。

### M3: 改善の不可逆性（ratchet-down）

一度改善した品質は二度と悪化しない。放置しても壊れない。
user のモチベーションに依存しない仕組み。

**具体例**: TAG_MISMATCH_BASELINE 51 → 48。
次にタグ不一致が増えたら CI が失敗する。元に戻ることはない。

### M4: 違反時の自己修復ガイド

formatViolationMessage が以下を全て出力する:
- 何がダメか（outdatedPattern）
- 何が正しいか（correctPattern）
- なぜそうすべきか（why + doc）
- 修正手順（migrationPath.steps）
- 例外条件（decisionCriteria.exceptions）

**効果**: 違反 → 修正のサイクルが短い。「なぜダメか分からない」が発生しない。

### M5: 因果分析が可能

relationships（dependsOn / enables）により、1 つの違反の波及範囲を辿れる。

**具体例**: AR-A1-DOMAIN（domain 純粋性）が壊れると、
AR-STRUCT-PURITY → AR-PATH-GROSS-PROFIT → AR-PATH-GROSS-PROFIT-CONSISTENCY
まで連鎖的に影響する。根本原因の特定が容易。

## 3. デメリット

### D1: 保守コストが高い

84 ルール × 7+ フィールド = 大量のメタデータ。

**リスク**: ルールの what / why / migrationPath が実態とずれたとき、
ルール自体が嘘になる。ドキュメントの陳腐化問題がルールに転移しただけとも言える。

**緩和策**:
- architectureRuleGuard がルールの整合性を自動検証（doc 実在チェック等）
- 例外圧検出がルールの妥当性を自動評価
- ルール変更時は CI が整合性を強制

**残存リスク**: what / why / migrationPath の内容の正確性は機械検証できない。
userが定期的に読んで確認する必要がある。

### D2: 過剰な制約は速度を殺す

84 ルールが全て gate だと、探索的な実装（プロトタイプ、実験）がやりにくい。

**緩和策**:
- maturity: 'experimental' + severity: 'warn' で新ルールは観察期間を設ける
- decisionCriteria.exceptions で正当な例外を明文化
- allowlist で個別例外を管理

**残存リスク**: 「まだルールがない新しいパターン」に対してシステムは何も言えない。
ルールの外で静かに進行する設計劣化は検出できない。

### D3: 偽の安心感

「ガードが全部通った = 設計が正しい」ではない。
ガードは定義されたパターンしか検出できない。

**具体例**: AR-TAG-CHART-VIEW は useMemo ≤ 4 を強制するが、
useMemo が 4 つでも全て不要な useMemo である可能性は検出できない。
数値的な制約は「質」を保証しない。

**緩和策**: ルールは「最低限の品質ゲート」であり、「最高の設計」ではないことを
自覚する。ルールを通過しても設計レビューの思考は省略しない。

### D4: 汎用性が低い

1 人 + AI 特化の設計であり、チーム開発にそのまま持ち込むと問題が生じる。

**チーム開発での問題**:
- 84 ルールの学習コストが新メンバーのオンボーディングを阻害
- PR レビューで十分な部分まで機械化しており、レビューアの判断力が育たない
- ルール変更の合意形成コストが高い（84 ルールの変更は全員に影響）

### D5: ルール数の肥大化リスク

現在 84。機能追加のたびにルールが増える傾向がある。

**予測**: 100 → 200 → 300 と増えると architectureRules.ts 自体が
God Object 化し、ルール間の矛盾や重複が発生する。

**緩和策**: 例外圧検出で不要ルールを特定、deprecated で段階的に削除、
ルールが 150 を超えたらカテゴリ別ファイル分割を検討。

## 4. リスク評価

| リスク | 深刻度 | 発生可能性 | 緩和策の有効性 | 残存リスク |
|---|---|---|---|---|
| ルール内容の陳腐化 | 高 | 高 | 中（機械検証は構造のみ） | **内容の正確性はuser 依存** |
| 過剰な制約による速度低下 | 中 | 中 | 高（experimental/warn） | 低 |
| 偽の安心感 | 中 | 中 | 低（本質的に限界） | **「質」の保証は不可能** |
| ルール数の肥大化 | 中 | 高 | 中（例外圧検出） | 150 超で分割が必要 |
| AI 依存度の上昇 | 低 | 高 | 低 | AI なしでの運用が困難に |

### 最大のリスク

**ルール自体が「正しい」という前提が崩れたとき、システム全体の信頼性が崩壊する。**

ルールの what / why / migrationPath が現実と乖離した場合、
AI はルールに従って「間違った修正」を自信を持って行う。
これは暗黙知の忘却よりも危険（間違った知識で行動する）。

**対策**: 定期的なルールレビュー（月 1 回程度）で内容の正確性を確認する。
例外圧が高いルールは優先的にレビューする。

### 本システムの独自性

1. **ルールが自己説明的**: 違反メッセージに what/why/correctPattern/migrationPath/decisionCriteria が全て含まれる
2. **因果グラフ**: ルール間の dependsOn/enables で波及分析が可能
3. **ratchet-down**: 改善の不可逆性を CI で保証
4. **maturity lifecycle**: experimental → stable → deprecated で硬直化を防止
5. **AI ファーストの設計**: 構造化データとして AI がパースしやすい形式

上記の 5 つを全て備えるシステムは、調査した限り先例がない。

## 6. 適用範囲の判断基準

### このシステムが有効な条件

- 開発者が 1-2 人（PR レビューの代替が必要）
- AI を積極的に活用する開発スタイル
- アーキテクチャが成熟している（頻繁な設計変更がない）
- ガードテストの文化が既にある（0 から構築するコストは高い）

### このシステムが不適切な条件

- チーム開発（3 人以上）— PR レビューのほうが効率的
- 設計が未成熟（スタートアップ初期等）— ルールが硬直化を招く
- AI を使わない開発 — ルールの構造化データの価値が半減
- ガードテストの文化がない — 0 からの構築コストが導入効果を上回る

## 7. 今後の進化方針

| 段階 | 状態 | 次のアクション |
|---|---|---|
| **現在** | 84 ルール / 全フィールド 100% | 運用しながら品質向上 |
| **ルール 100 超** | カテゴリ別分割を検討 | architectureRules/ ディレクトリ化 |
| **ルール 150 超** | ルール間の矛盾検出が必要 | conflicts の自動検証 |
| **チーム化** | ルールの学習コスト対策 | ルールの優先度付け（必読 / 参照 / 詳細） |

### カテゴリ分割の設計方針（100 ルール超で実施）

現在の 84 ルールは 1 ファイル（architectureRules.ts）に集約されている。
100 ルールを超えた時点で以下の構成に分割する:

```
app/src/test/architectureRules/
├── index.ts              ← 型定義 + lookup 関数 + 全ルール統合 export
├── layerBoundary.ts      ← AR-A1-* (層境界)
├── codePattern.ts        ← AR-G4, C3, G3, E4, C5, G2 (コードパターン)
├── size.ts               ← AR-G5-*, G6, C6 (サイズ・複雑度)
├── path.ts               ← AR-PATH-* (正本取得経路)
├── structure.ts          ← AR-STRUCT-* (構造ガード)
├── tag.ts                ← AR-TAG-* (責務タグ閾値)
├── canon.ts              ← AR-CANON-* (正本化)
└── noNewDebt.ts          ← AR-001〜005 (新規負債禁止)
```

**分割の原則**:
- 各ファイルが `readonly ArchitectureRule[]` を export する
- `index.ts` が全ファイルを集約して `ARCHITECTURE_RULES` を構築する
- lookup 関数（getRuleById 等）は index.ts に残る
- ガードテストは引き続き index.ts から import する（参照元は変更不要）
- 分割はルールの `id` プレフィックスに自然に対応する

### 定期メンテナンス

- **月 1 回**: 例外圧が高いルールの内容レビュー
- **機能追加時**: 新パターンは experimental で追加
- **アーキテクチャ変更時**: 影響を受けるルールを deprecated → 新ルール追加
