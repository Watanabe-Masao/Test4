# WID-040 — サマリーKPI（原価明細）

## 1. 概要

`ctx.costDetailData` が非 null のとき、`KpiGrid` + 4 枚の `KpiCard`（タイプ入/出 + 原価算入費合計 + 原価算入率）を registry 行の render 関数内に **inline で JSX 展開**する Unified 版 widget（CostDetail/widgets.tsx:17-56）。

## 2. Data Acquisition Path

- `ctx.costDetailData` を変数 `d` に束縛して全データを抽出（`acquisitionPath: ctx-direct`）
- `fmtCurrency` を `ctx` から分割代入、`formatPercent` を `@/domain/formatting` から直接 import
- `onExplain` は `ctx.onExplain('totalCostInclusion')` として metric ID 固定で呼び出し
- render 内で早期 return: `if (!d?.typeIn || !d?.typeOut) return null`

## 3. Context Fields Consumed — 解説

- `UnifiedWidgetContext.costDetailData: CostDetailData` は **page-local optional field**（`components/widgets/types.ts:214` のソース注記「CostDetail 固有」、全ページで optional）
- CostDetail ページ上でのみ実体を持ち、他ページでは undefined。isVisible predicate で実体有無を gate している
- `fmtCurrency` は core required field（`components/widgets/types.ts:82`）、通貨単位設定連動
- `onExplain` は core required field（`:77`）、metric 説明 modal の起動 callback
- touch している field 8 個はすべて `costDetailData` の sub-field（2 階層深度）

## 4. Composition

- 子 component は `KpiGrid` + `KpiCard` のみ（いずれも `@/presentation/components/common/tables` から import）
- `palette` 色トークンを直接指定（`palette.blueDark` / `palette.dangerDark` / `palette.orange` / `palette.orangeDark`）
- render 関数が JSX を inline で構築しており、子への委譲ではなく**widget 自身が layout を所有**
- WID-001 / WID-002 と異なり full ctx の child への forward は無い

## 5. Visibility

- 条件: `(ctx) => ctx.costDetailData != null`（widgets.tsx:56）
- `costDetailData` は Unified ctx で optional（他ページでは undefined）。CostDetail page 以外では非可視
- `isVisible` 通過後、render 内で更に `if (!d?.typeIn || !d?.typeOut) return null` の**二重 null check**

## 6. Invariants

- 表示される `costInclusionRate` は `totalCostInclusionAmount / totalSales` と整合する想定（`CostDetailData` 正本側で保証）
- `typeIn` / `typeOut` の `cost` と `price` の関係は CostDetailData 正本側で定義（本 widget は表示のみ）
- `formatPercent` は小数第 2 位（CLAUDE.md コーディング規約）

## 9. Pipeline Concerns / Upstream Requests

> 事実のみを記録する。widget 側 workaround / 「〜すべき」は書かない

- **`costDetailData` が Unified ctx の page-local optional**: `UnifiedWidgetContext.costDetailData?: CostDetailData`（`types.ts:214`）は **CostDetail page 専用**の field を Unified context に optional として配置している。Unified を名乗りつつ page 結合している非対称性（`projects/completed/architecture-debt-recovery/inquiry/01-widget-registries.md` §「ambiguity §b」および `plan.md` の Purpose 冒頭で指摘される「page-coupled ctx」に該当）。該当責務層: application（page-local field を Unified ctx から分離し `useXxxPlan` 経由に移管する可否）
- **二重 null check（isVisible + render 内）**: `isVisible: ctx.costDetailData != null` を通過しても、render 内で `if (!d?.typeIn || !d?.typeOut) return null` を追加で必要とする。isVisible の保証レベルと render の前提が一致していない。該当責務層: application（`CostDetailData` 型の required field 保証、または isVisible predicate の強化）
- **metric ID `'totalCostInclusion'` の presentation 層ハードコード**: `onExplain('totalCostInclusion')` が registry 行にリテラル文字列として埋め込まれている。metric ID の正本は domain / application にあるはずだが、widget 層で直接文字列参照している。該当責務層: domain（`MetricId` 型での type-safe 参照化の可否）
- **render 関数が inline JSX + 色 token を所有**: `palette.blueDark` / `palette.dangerDark` / `palette.orange` / `palette.orangeDark` の 4 色が registry 行に直接記述されており、widget 登録層に design decision が漏れている。該当責務層: presentation（chart-semantic-colors 契約や意味トークン化の可否検証、`references/02-design-system/docs/chart-semantic-colors.md` 参照）
- **`typeLabel` の template リテラル結合**: `` `${d.typeLabel}入` `` / `` `${d.typeLabel}出` `` の文字列結合が widget 層で実行されている。label 構築責務が domain / application にあれば presentation は受け取るだけで済むが、現状は presentation で組み立てている。該当責務層: domain または application（label 構築責務の位置検証）
- **Unified `WidgetDef` 型で registry 定義されているが実体は CostDetail page 専用**: WidgetDef 型 B（`contextType: Unified`）を使用しているため、理論上は他ページで render されうるが `isVisible` で強制 filter されている。本来ページ固有 widget なら Dashboard-local 相当の「page-local `WidgetDef`」で定義する方が意味的に整合する可能性がある（inquiry/01 §「WidgetDef の 2 型」の型 A / B の使い分け基準の揺れ）。該当責務層: application + presentation（widget 型と page 帰属の整理）
