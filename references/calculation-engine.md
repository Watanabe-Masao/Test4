# 計算エンジン仕様書

## 1. 概要

本計算エンジンは、小売業における仕入・粗利管理のコアロジックを提供する。

### モジュール構成

| モジュール | ファイル | 役割 |
|-----------|---------|------|
| 在庫法 | `invMethod.ts` | 実績粗利計算 |
| 推定法 | `estMethod.ts` | 在庫推定指標計算 |
| 予算分析 | `budgetAnalysis.ts` | 予算消化・達成率分析 |
| 予測 | `forecast.ts` | 売上予測・異常値検出 |
| 全店集計 | `aggregation.ts` | 複数店舗の集約ロジック |
| 売変影響分析 | `discountImpact.ts` | 売変ロス原価算出 |
| ピン止め区間 | `pinIntervals.ts` | 区間別在庫法粗利率計算 |
| 要因分解 | `factorDecomposition.ts` | シャープリー値ベースの売上変動要因分解 |
| 生データ集約 | `rawAggregation.ts` | DuckDB 生データの JS 集約（累積・移動平均・統計） |
| ユーティリティ | `utils.ts` | 安全除算・数値変換・フォーマット |

### 設計原則

- **全て純粋関数**: 副作用なし、入力から出力を決定的に算出
- **フレームワーク非依存**: React/DOM への依存なし、単体テスト容易
- **配置**: `domain/calculations/` ディレクトリに集約
- **統合フロー**: `CalculationOrchestrator`（`application/services/`）が各モジュールを統合し、`dailyBuilder` による日次レコード構築、`storeAssembler` による `StoreResult` 組み立て、`aggregateResults` による全店集約を制御する

## 2. 在庫法 (invMethod.ts) -- 実績粗利

### スコープ

全売上・全仕入（花・産直を含む全体）

### 計算式

```
売上原価 = 期首在庫 + 総仕入高 - 期末在庫
粗利益   = 売上高 - 売上原価
粗利率   = 粗利益 / 売上高
```

### 入力 (`InvMethodInput`)

| フィールド | 型 | 説明 |
|-----------|---|------|
| `openingInventory` | `number \| null` | 期首在庫（実績値） |
| `closingInventory` | `number \| null` | 期末在庫（実績値） |
| `totalPurchaseCost` | `number` | 総仕入原価 |
| `totalSales` | `number` | 総売上高 |

### 出力 (`InvMethodResult`)

| フィールド | 型 | 説明 |
|-----------|---|------|
| `cogs` | `number \| null` | 売上原価 |
| `grossProfit` | `number \| null` | 粗利益 |
| `grossProfitRate` | `number \| null` | 粗利率 |

### 必要データ

- 期首在庫、期末在庫（実績値）。いずれかが `null` の場合は計算不可（全結果が `null`）

### 用途

実際の損益確認。会計的な粗利を正確に算出する。

## 3. 推定法 (estMethod.ts) -- 在庫推定指標

### スコープ

在庫販売のみ（花・産直除外）

### 計算式

```
コア売上 = 総売上 - 花売価 - 産直売価
粗売上   = コア売上 / (1 - 売変率)
推定原価 = 粗売上 × (1 - 値入率) + 原価算入費（costInclusionCost）
推定粗利 = コア売上 - 推定原価
推定粗利率 = 推定粗利 / コア売上
推定期末在庫 = 期首在庫 + 期中仕入原価(在庫販売分) - 推定原価
```

### 入力 (`EstMethodInput`)

| フィールド | 型 | 説明 |
|-----------|---|------|
| `coreSales` | `number` | コア売上（花・産直・売上納品除外） |
| `discountRate` | `number` | 売変率 |
| `markupRate` | `number` | 値入率 |
| `costInclusionCost` | `number` | 原価算入費（消耗品費） |
| `openingInventory` | `number \| null` | 期首在庫（在庫販売分） |
| `inventoryPurchaseCost` | `number` | 期中仕入原価（在庫販売分、花・産直除外） |

### 出力 (`EstMethodResult`)

| フィールド | 型 | 説明 |
|-----------|---|------|
| `grossSales` | `number` | 粗売上（売変前売価） |
| `cogs` | `number` | 推定原価 |
| `margin` | `number` | 推定マージン（実粗利ではない） |
| `marginRate` | `number` | 推定マージン率（実粗利率ではない） |
| `closingInventory` | `number \| null` | 推定期末在庫 |

### 補助関数

- **`calculateCoreSales(totalSales, flowerSalesPrice, directProduceSalesPrice)`**: コア売上を算出。コア売上がマイナスになる場合（売上納品超過）は `0` にクランプし、超過フラグ・超過額を返す
- **`calculateDiscountRate(salesAmount, discountAmount)`**: 売変率 = 売変額 / (売上 + 売変額)

### 用途

在庫差異検知、異常検出。推定在庫と実績在庫を比較し、見えない損失・異常を発見する。

> **重要**: 推定法は実粗利ではない。在庫推定の基礎指標であり、推定在庫と実績在庫の乖離を検知するために使用する。

## 4. 在庫法と推定法の違い（最重要）

この2つの計算手法は**目的が根本的に異なる**。混同すると経営判断を誤る原因となるため、違いを正確に理解すること。

| | 在庫法 | 推定法 |
|---|---|---|
| **本質** | 損益計算 | 在庫推定 |
| **対象** | 全売上・全仕入（花・産直含む） | 在庫販売のみ（花・産直除外） |
| **算出根拠** | 実績在庫増減 | 値入率・売変率からの逆算 |
| **必要データ** | 期首/期末在庫（実績値） | 値入率・売変率 |
| **結果の性質** | 会計的に正確な実績粗利 | 推定値（理論上の在庫水準） |
| **利用場面** | 実際の損益確認 | 在庫差異検知・異常検出 |

### 使い分けの指針

- **「今月いくら儲かったか？」** → 在庫法（`invMethod`）
- **「在庫は帳簿通りか？おかしな損失はないか？」** → 推定法（`estMethod`）

## 5. 売変影響分析 (discountImpact.ts)

### 計算式

```
売変ロス原価 = (1 - 値入率) × コア売上 × 売変率 / (1 - 売変率)
```

### 入力 (`DiscountImpactInput`)

| フィールド | 型 | 説明 |
|-----------|---|------|
| `coreSales` | `number` | コア売上 |
| `markupRate` | `number` | 値入率 |
| `discountRate` | `number` | 売変率 |

### 出力 (`DiscountImpactResult`)

| フィールド | 型 | 説明 |
|-----------|---|------|
| `discountLossCost` | `number` | 売変ロス原価 |

### 意味

売変（値引）によって失われた売価を、原価換算した金額。値引がどれだけの原価相当の損失を生んでいるかを可視化する。

## 6. 予算分析 (budgetAnalysis.ts)

### 計算項目

| 項目 | 計算式 |
|------|--------|
| 予算達成率 | 売上 / 予算 |
| 予算消化率 | 売上 / 経過日までの累計予算（日別累計 vs 予算累計） |
| 予算経過率 | 経過日までの累計予算 / 月間予算 |
| 日平均売上 | 売上 / 営業日数（営業日ベース） |
| 月末予測 | 実績売上 + 日平均売上 × 残日数 |
| 達成率予測 | 月末予測売上 / 予算 |
| 残余予算 | 予算 - 売上 |
| 日別累計 | 日ごとの売上累計・予算累計の `Map` |

### 入力 (`BudgetAnalysisInput`)

| フィールド | 型 | 説明 |

| `salesDaily` | `ReadonlyMap<number, number>` | 日別売上 |
| `elapsedDays` | `number` | 経過日数 |
| `salesDays` | `number` | 営業日数（売上がある日数） |
| `daysInMonth` | `number` | 月の日数 |

### 出力 (`BudgetAnalysisResult`)

| フィールド | 型 | 説明 |
|-----------|---|------|
| `budgetAchievementRate` | `number` | 予算達成率 |
| `budgetProgressRate` | `number` | 予算消化率 |
| `budgetElapsedRate` | `number` | 予算経過率 |
| `averageDailySales` | `number` | 日平均売上 |
| `projectedSales` | `number` | 月末予測売上 |
| `projectedAchievement` | `number` | 予算達成率予測 |

| `dailyCumulative` | `ReadonlyMap<number, { sales, budget }>` | 日別累計 |

## 7. 予測・異常値検出 (forecast.ts)

### 月間カレンダー

- **月曜始まり**のカレンダー形式で週を区切る
- `getWeekRanges(year, month)` が週番号・開始日・終了日を返す

### 週単位集計 (`calculateWeeklySummaries`)

各週について以下を算出:
- 週間売上合計
- 週間粗利合計
- 週間粗利率
- 営業日数（売上 > 0 の日数）

### 曜日別平均 (`calculateDayOfWeekAverages`)

- 曜日ごと（0=日, 1=月, ... 6=土）の平均売上を算出
- 売上がある日のみカウント

### 異常値検出 (`detectAnomalies`)

- 日別売上の平均と標準偏差を計算
- Zスコア = (値 - 平均) / 標準偏差
- デフォルト閾値: `|Zスコア| > 2.0` で異常値判定
- データが3件未満、または標準偏差が0の場合は検出をスキップ

### 統合関数 (`calculateForecast`)

`ForecastInput` を受け取り、週別サマリー・曜日別平均・異常値検出結果をまとめて返す。

## 8. 全店集計 (aggregation.ts)

### 集計ルール

| 項目種別 | 集計方法 | 関数 |
|---------|---------|------|
| 金額項目 | 単純合計 | `sumStoreValues` |
| nullable金額項目 | 単純合計（全てnullなら null） | `sumNullableValues` |
| 率項目 | 売上高加重平均 | `weightedAverageBySales` |

### 加重平均の計算

```
加重平均 = Σ(store.rate × store.sales) / Σ(store.sales)
```

- 売上が `> 0` の店舗のみ加重対象
- 分母（総売上）が `0` の場合 → `0` を返す

## 9. ピン止め区間計算 (pinIntervals.ts)

### 概要

期中に棚卸を行った日（ピン止め日）を基に、区間ごとの在庫法粗利率を算出する。

### 入力

- `daily`: 日次レコード（`Map<number, DailyRecord>`）
- `openingInventory`: 期首在庫
- `pins`: `[day, closingInventory][]` -- ピン止め日と期末在庫の配列（日付昇順）

### 区間ごとの計算

各区間（前回ピン止め日の翌日 〜 当該ピン止め日）について:
- 区間売上 = 区間内の日次売上合計
- 区間仕入原価 = 区間内の日次仕入原価合計
- 売上原価 = 前区間の期末在庫 + 区間仕入原価 - 当区間の期末在庫
- 粗利益 = 区間売上 - 売上原価
- 粗利率 = 粗利益 / 区間売上

## 10. 売上要因分解 (factorDecomposition.ts)

### 概要

前年比較における売上変動を、シャープリー値（ゲーム理論の公理的手法）を用いて複数要因に分解する。
順序非依存・完全一致・交互作用の公平配分が数学的に保証される。

### 分解レベル

| レベル | 変数 | 要因 | 関数 |
|--------|------|------|------|
| 2要素 | C, T | 客数効果 + 客単価効果 | `decompose2` |
| 3要素 | C, Q, P̄ | 客数効果 + 点数効果 + 単価効果 | `decompose3` |
| 4要素 | C, Q, p, s | 客数効果 + 点数効果 + 価格効果 + 構成比変化効果 | `decompose5` |

### 2要素シャープリー分解 (n=2)

```
S = C × T  (客数 × 客単価)
φ_C = ΔC × ½(T₀ + T₁)
φ_T = ΔT × ½(C₀ + C₁)
```

### 3要素シャープリー分解 (n=3)

```
S = C × Q × P̄  (客数 × 一人当たり点数 × 点単価)
重み = |S|!(n-1-|S|)!/n! → (2/6, 1/6, 1/6, 2/6)

φ_C = ΔC/6 × (2Q₀P̄₀ + Q₁P̄₀ + Q₀P̄₁ + 2Q₁P̄₁)
φ_Q = ΔQ/6 × (2C₀P̄₀ + C₁P̄₀ + C₀P̄₁ + 2C₁P̄₁)
φ_P̄ = ΔP̄/6 × (2C₀Q₀ + C₁Q₀ + C₀Q₁ + 2C₁Q₁)
```

### 完全4変数シャープリー分解 (n=4)

```
S = C × Q × (p·s)
4つの状態単価: P₀₀=p₀·s₀, P₁₀=p₁·s₀, P₀₁=p₀·s₁, P₁₁=p₁·s₁
重み: k=0→¼, k=1→1/12, k=2→1/12, k=3→¼

φ_C = ΔC × [¼Q₀P₀₀ + 1/12(Q₁P₀₀+Q₀P₁₀+Q₀P₀₁) + 1/12(Q₁P₁₀+Q₁P₀₁+Q₀P₁₁) + ¼Q₁P₁₁]
φ_Q = ΔQ × [¼C₀P₀₀ + 1/12(C₁P₀₀+C₀P₁₀+C₀P₀₁) + 1/12(C₁P₁₀+C₁P₀₁+C₀P₁₁) + ¼C₁P₁₁]
φ_p = Δp·(A₀s₀ + A₁s₁)
φ_s = Δs·(A₀p₀ + A₁p₁)
  A₀ = ¼C₀Q₀ + 1/12(C₁Q₀ + C₀Q₁ + C₁Q₁)
  A₁ = 1/12(C₀Q₀ + C₁Q₀ + C₀Q₁) + ¼C₁Q₁
```

### 価格/構成比分解 (`decomposePriceMix`)

カテゴリ別データがない場合の代替として、平均単価変動を2変数シャープリーで分解:

```
φ_p^(P̄) = ½[Σ(p₁ᵢ−p₀ᵢ)s₀ᵢ + Σ(p₁ᵢ−p₀ᵢ)s₁ᵢ]  (価格効果)
φ_s^(P̄) = ½[Σp₀ᵢ(s₁ᵢ−s₀ᵢ) + Σp₁ᵢ(s₁ᵢ−s₀ᵢ)]  (構成比変化効果)
```

### 性質（数学的保証）

- **効率性**: φ_C + φ_Q + φ_p + φ_s = ΔS（完全一致）
- **対称性**: 同じ限界貢献を持つ変数には同じ値が配分される
- **順序非依存**: 分解の計算順序に依存しない
- **交互作用の公平配分**: 複数変数が同時に変化した場合の相互作用を公平に配分

## 11. ユーティリティ (utils.ts)

### safeDivide(numerator, denominator, fallback = 0)

ゼロ除算を防止する安全な除算。`denominator` が `0` の場合は `fallback` を返す。

### safeNumber(n)

`null`, `undefined`, `NaN` を `0` に変換する。型不明の値を安全に数値化する。

### フォーマット関数

| 関数 | 説明 | 例 |
|------|------|---|
| `formatCurrency(n)` | 四捨五入 + カンマ区切り | `1234567` → `"1,234,567"` |
| `formatManYen(n)` | 万円表示 | `12340000` → `"+1234万円"` |
| `formatPercent(n, decimals)` | パーセント表示 | `0.2534` → `"25.34%"` |
| `formatPointDiff(n, decimals)` | ポイント差表示 | `0.015` → `"+1.5pt"` |

いずれも `null` / `NaN` の場合は `"-"` を返す。

## 12. エッジケース処理

### ゼロ除算ガード

全ての除算に `safeDivide` を適用。粗利率、予算達成率、加重平均など、分母がゼロになりうる全箇所で保護される。

### 期首/期末在庫の欠如

在庫法（`invMethod`）: `openingInventory` または `closingInventory` が `null` の場合、全計算結果を `null` で返す。

### 売上納品超過

`calculateCoreSales`: コア売上がマイナスになる場合（売上納品が総売上を上回る異常ケース）:
- コア売上を `0` にクランプ
- `isOverDelivery = true` フラグを設定
- 超過額 `overDeliveryAmount` を記録

### Null/NaN 統一処理

- `safeNumber`: 型不明の値を安全に `number` へ変換（`null`/`undefined`/`NaN` → `0`）
- フォーマット関数: `null`/`NaN` → `"-"` 表示

### 売変率が1の場合

推定法（`estMethod`）: 売変率が `1`（全額値引）の場合、除数 `(1 - 売変率)` が `0` となるため、コア売上をそのまま粗売上として使用する。

### 売変影響分析の除数ガード

`discountImpact`: `(1 - 売変率)` が `0` 以下の場合、除数を `1` に置換して計算を継続する。

### 異常値検出の最小データ要件

`detectAnomalies`: データが3件未満、または標準偏差が `0` の場合は空配列を返す。


## 13. 追加モジュール

### 13.7 生データ集約 (rawAggregation.ts)

DuckDB から取得した生データ（`store_day_summary` の `SELECT * WHERE` 結果）を
JS 純粋関数で集約する。SQL ウィンドウ関数の代替として使用。

| 関数 | 説明 | 代替する SQL |
|---|---|---|
| `aggregateByDay` | 日別に売上・客数・仕入を集約 | `GROUP BY date_key` |
| `cumulativeSum` | 日別累積売上を算出 | `SUM() OVER (ORDER BY date_key)` |
| `movingAverage` | N日移動平均 | `AVG() OVER (ROWS N PRECEDING)` |
| `dowAggregate` | 曜日別平均売上（dateKey から曜日判定） | `AVG(sales) GROUP BY dow` |
| `hourlyAggregate` | 時間帯別集約 | `SUM() GROUP BY hour` |
| `aggregatePeriodRates` | 期間集約（平均値入率・売変率） | `AVG() GROUP BY` |
| `stddevPop` | 母標準偏差 | `STDDEV_POP()` |
| `zScore` | Zスコア算出 | SQL 内 `(value - AVG) / STDDEV` |
| `coefficientOfVariation` | 変動係数（CV = σ/μ） | SQL 内算出 |
| `rankBy` | 降順ランキング | `RANK() OVER (ORDER BY ...)` |
| `yoyMerge` | 当年/前年のキーマッチング結合 | `FULL OUTER JOIN` |
| `categoryShare` | 構成比算出（各要素 / 合計） | `value / SUM() OVER ()` |

**設計原則:**
- 全て純粋関数（副作用なし、フレームワーク非依存）
- 23テスト（`rawAggregation.test.ts`）で全関数をカバー
- DuckDB の生データ型（`store_day_summary` のカラム）に依存しない汎用設計

# 付録: 計算式ユーザー解説

本アプリケーションで使用される計算式の解説です。

---

## 1. 粗利計算

### 在庫法（棚卸法）

最も基本的な粗利計算方法です。実際の棚卸在庫から売上原価を逆算します。

```
売上原価 = 期首在庫 + 総仕入高 - 期末在庫
粗利益   = 売上高 - 売上原価
粗利率   = 粗利益 ÷ 売上高
```

**使い方**: 「今月いくら儲かったか？」を知りたいときに使います。

**必要データ**: 期首在庫（月初）、期末在庫（月末）、仕入データ、売上データ

### 推定法

在庫法とは目的が異なります。棚卸せずに「理論上の在庫」を推定し、実際の在庫との差から異常を検出します。

```
コア売上   = 総売上 - 花売価 - 産直売価
粗売上     = コア売上 ÷ (1 - 売変率)
推定原価   = 粗売上 × (1 - 値入率) + 消耗品費
推定在庫   = 期首在庫 + 仕入原価 - 推定原価
```

**使い方**: 「在庫は帳簿通りか？おかしな損失（万引き・破損・記帳ミス）はないか？」を調べるときに使います。

予算消化率 = 売上 ÷ 経過日までの累計予算

## 3. 売上変動の要因分解（シャープリー値）

### なぜシャープリー値を使うのか？

売上が前年から変動した原因を分析するとき、「客数が増えた」「客単価が上がった」と単純に分けると、**分解の順序によって結果が変わる**という問題があります。

例えば、客数が 100→110人、客単価が 1,000→1,200円に変化した場合：

- 先に客数を計算: 客数効果 = 10人 × 1,000円 = **10,000円**
- 先に客単価を計算: 客数効果 = 10人 × 1,200円 = **12,000円**

どちらも間違いではありませんが、**順序によって結果が変わってしまう**のは問題です。

**シャープリー値**は、ゲーム理論で「貢献の公平な配分」を数学的に定義した手法で、以下を保証します：

- **順序非依存**: どの要因から計算しても結果が同じ
- **完全一致**: 全要因の効果を合計すると、売上差額と完全に一致
- **公平配分**: 複数要因が同時に変化した場合の「交互作用」を公平に配分

### 2要素分解（客数 × 客単価）

売上 = 客数(C) × 客単価(T) の2変数で分解します。

```
客数効果   = (当年客数 - 前年客数) × ½(前年客単価 + 当年客単価)
客単価効果 = (当年客単価 - 前年客単価) × ½(前年客数 + 当年客数)
```

**読み方**: 客数効果は「客数の変化が売上にどれだけ影響したか」を示します。前年と当年の客単価の平均を使うことで、交互作用を公平に配分しています。

### 3要素分解（客数 × 点数 × 単価）

売上 = 客数(C) × 一人当たり点数(Q) × 点単価(P̄) の3変数で分解します。

n=3のシャープリー重みは (2/6, 1/6, 1/6, 2/6) となり、各変数の効果は他の2変数の全組み合わせに対する限界貢献の加重平均です。

```
客数効果 = (C₁-C₀)/6 × (2Q₀P₀ + Q₁P₀ + Q₀P₁ + 2Q₁P₁)
点数効果 = (Q₁-Q₀)/6 × (2C₀P₀ + C₁P₀ + C₀P₁ + 2C₁P₁)
単価効果 = (P₁-P₀)/6 × (2C₀Q₀ + C₁Q₀ + C₀Q₁ + 2C₁Q₁)
```

**読み方**: 各効果は「その変数だけが変化したとき、他の変数が前年・当年のどちらの値をとるかの全パターンの加重平均」です。

### 4要素分解（客数 × 点数 × 価格 × 構成比）

3要素の「単価効果」をさらに「価格効果」と「構成比変化効果」に分解します。

- **価格効果**: 各カテゴリの単価変動による影響（値上げ・値下げ）
- **構成比変化効果**: 高単価品・低単価品の販売比率変化による影響（商品ミックスの変化）

4変数(C, Q, p, s)の完全シャープリー値を使用し、4!=24通りの全順序の限界貢献の加重平均を閉形式で計算します。

**4つの状態単価**:
- P₀₀ = 前年価格 × 前年構成比（前年の平均単価）
- P₁₁ = 当年価格 × 当年構成比（当年の平均単価）
- P₁₀ = 当年価格 × 前年構成比（価格だけ変化した場合の単価）
- P₀₁ = 前年価格 × 当年構成比（構成比だけ変化した場合の単価）

n=4のシャープリー重みは (1/4, 1/12, 1/12, 1/4) で、各変数の効果は残り3変数の全部分集合（8パターン）に対する限界貢献の加重平均です。

#### 具体例

前年: 部門A 100個×1,000円、部門B 100個×500円 → 売上 150,000円
当年: 部門A 150個×1,100円、部門B 50個×500円 → 売上 190,000円

売上差 = +40,000円 を4要素に分解すると：

| 要因 | 効果 | 意味 |
|------|------|------|
| 客数効果 | ... | 来店客数の変化による影響 |
| 点数効果 | ... | 一人当たり購入点数の変化 |
| 価格効果 | +... | 部門Aの値上げ（1,000→1,100円）の影響 |
| 構成比変化効果 | +... | 高単価品Aの構成比増加の影響 |
| **合計** | **+40,000円** | **売上差額と完全一致** |

### ウォーターフォールチャートの見方

ダッシュボードのウォーターフォールチャートは、左端の「前年売上」から右端の「当年売上」への変動を、各要因の積み上げで表現します。

- **緑色の棒**: プラス要因（売上を押し上げた要因）
- **赤色の棒**: マイナス要因（売上を押し下げた要因）
- **青紫色の棒**: 前年・当年の売上合計

上部の切替ボタンで「2要素（客数・客単価）」「3要素（客数・点数・単価）」「4要素（+価格・構成比）」を選択できます。

## 4. 売変影響分析

値引（売変）によって失われた金額を原価換算した指標です。

```
売変ロス原価 = (1 - 値入率) × コア売上 × 売変率 ÷ (1 - 売変率)
```

**読み方**: 値引がどれだけの「仕入原価相当の損失」を生んでいるかを示します。値引率が高い部門では、実際の粗利への影響が大きくなります。

---

## 5. 異常値検出

日別売上の標準偏差を使い、通常パターンから大きく外れた日を自動検出します。

```
Zスコア = (その日の売上 - 平均売上) ÷ 標準偏差
```

Zスコアの絶対値が 2.0 を超える日が「異常値」として検出されます。

**活用例**: 特売日の効果測定、天候・イベントの影響把握、データ入力ミスの発見

---

## 6. 全店集計のルール

複数店舗のデータを集計する際のルール：

| 項目 | 集計方法 | 理由 |
|------|---------|------|
| 金額（売上、仕入等） | 単純合計 | 金額は足し算が自然 |
| 率（粗利率、達成率等） | 売上高加重平均 | 単純平均では売上規模の違いを反映できない |

**加重平均の計算**:
```
全店粗利率 = Σ(各店の粗利率 × 各店の売上) ÷ Σ(各店の売上)
```

例: A店（売上1,000万、粗利率30%）、B店（売上100万、粗利率50%）の場合
→ 全店粗利率 = (1000×0.30 + 100×0.50) ÷ (1000+100) = 31.8%（単純平均40%ではない）

## 7. ピン止め区間分析

月中に棚卸（ピン止め）を行った場合、その前後の区間ごとに粗利率を計算できます。

```
区間の売上原価 = 区間の期首在庫 + 区間の仕入 - 区間の期末在庫
区間の粗利率   = (区間の売上 - 区間の売上原価) ÷ 区間の売上
```

**活用例**: 特売期間の前後で粗利率がどう変化したかを分析
