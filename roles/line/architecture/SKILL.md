# architecture — スキル（論理構造 + 方法論）

## SKILL-1: レイヤー影響分析

### 論理構造（なぜこの手順か）

- import の方向に違反すると → guards/layerBoundaryGuard.test.ts が落ちる → CI が FAIL する
- フレームワーク依存が domain/ に入ると → 純粋関数テストにモックが必要になる → テストの信頼性が落ちる
- 許可リストに安易に追加すると → 境界の保護が弱まる → 依存方向が徐々に崩壊する

### 方法論（手順）

1. 変更対象のファイルパスを特定する
2. 各ファイルが属するレイヤーを判定する（domain / application / infrastructure / presentation）
3. 変更による新しい import が既存の依存方向に違反しないか確認する
4. 違反する場合、以下を検討:
   - インターフェース（ポート）経由のアクセスに変更できるか
   - 許可リストに追加する正当な理由があるか
   - ファイルの配置レイヤーを変更すべきか

## SKILL-2: ファイル分割判断

### 論理構造（なぜこの手順か）

- styles/ロジック/JSX が混在すると → スタイル変更がビジネスロジックに影響する → カスケード故障のリスク
- 300行超のファイルは → 複数の変更理由を持つ可能性が高い → 1つの変更が意図しない箇所を壊す
- 分割しすぎると → ファイル間の跳躍が増え → 読みやすさが低下する（300行は閾値であって義務ではない）

### 方法論（手順）

1. **300行以下**: 分割不要
2. **300〜500行**: 変更頻度が異なる部分があれば分割を検討
3. **500行超**: 原則分割（設計思想4「変更頻度が異なるものは分離する」）

### 分割パターン

| パターン | 分割先 | 適用条件 |
|---|---|---|
| styled-components + ロジック + JSX | `*.styles.ts` + `use*Data.ts` + `*.tsx` | UI コンポーネント |
| 複数の export function | 責務別ファイル | ユーティリティ |
| クエリ定義 + フック | `queries/*.ts` + `hooks/duckdb/*.ts` | DuckDB 関連 |

## SKILL-3: 許可リスト管理

### 論理構造（なぜこの手順か）

- 削除済みファイルが許可リストに残ると → INV-ARCH-06 が FAIL する → 許可リストの信頼性が低下する
- 理由なく追加すると → 「なぜこの例外が許されているか」が不明になる → 将来の判断が困難になる
- review-gate に理由を伝えないと → レビュー時に許可リスト変更が見落とされる

### 方法論（手順）

#### 追加

1. 追加理由を文書化する（「なぜ infrastructure に直接依存する必要があるか」）
2. `guards/layerBoundaryGuard.test.ts` の対応する許可リストにファイルパスを追加
3. `npm test` で全テストが通ることを確認
4. review-gate に理由を伝達

#### 削除

1. 対象ファイルが削除済みまたは依存が解消済みであることを確認
2. 許可リストからファイルパスを削除
3. `npm test` で INV-ARCH-06（ファイル実在確認）を含む全テストが通ることを確認
