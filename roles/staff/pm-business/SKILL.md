# pm-business — スキル（論理構造 + 方法論）

## SKILL-1: タスク分解

### 論理構造（なぜこの手順か）

- ロール別に分解しないと → 責務が曖昧になる → 越境が発生し品質が低下する
- 受入基準が測定不能だと → 完了の判定ができない → タスクが永遠に「進行中」になる
- 作業順序を決めないと → 依存関係のあるタスクが並列で走る → 手戻りが発生する

### 方法論（手順）

1. タスクの内容を理解する
2. CLAUDE.md §ルーティング表 で主ロールと連携先を特定する
3. タスク規模を判定する（Small / Medium / Large）
4. 受入基準を定義する（測定可能な条件にする）
5. 作業項目と順序を決定する
6. タスク分解書を作成する
7. タスクに適切なロールのエージェントを割り当てる（SKILL-3 参照）

### 出力テンプレート

```
## タスク分解書

### タスク概要
（人間からの依頼内容）

### タスク規模
Small / Medium / Large

### 受入基準
- [ ] （測定可能な条件1）
- [ ] （測定可能な条件2）
- [ ] CI 5段階ゲート通過

### 作業分解
1. [architecture] （設計判断の内容）← Large の場合
2. [implementation] （実装内容）
3. [invariant-guardian] （不変条件の確認）← 計算変更の場合
4. [review-gate] 禁止事項チェック + CI 確認
5. [documentation-steward] ドキュメント更新 ← 必要な場合

### 連携順序
architecture → implementation → review-gate
```

## SKILL-2: 完了判定

### 論理構造（なぜこの手順か）

- review-gate が PASS でも受入基準を満たさない場合がある → 品質は通ったが要件は未達
- 受入基準の照合を省くと → 「動くけど要件と違う」コードが完了扱いになる
- 不足がある場合に追加指示しないと → 不完全な状態で次のタスクに進む → 技術的負債が蓄積する

### 方法論（手順）

1. review-gate の結果（PASS / FAIL）を確認する
2. PASS の場合、受入基準の各項目が満たされているか確認する
3. 全項目 OK → 完了を pm-business から人間に報告
4. 不足がある場合 → implementation に追加作業を連携指示する

## SKILL-3: タスクへのエージェント割り当て

### 論理構造（なぜこの手順か）

- pm-business（指示者）をスキップすると → タスク分解・受入基準が不在のまま作業が始まる → 完了判定できない
- タスクにエージェントを割り当てないと → 誰が何をやるか不明 → 責務の越境・重複が発生する
- エージェントに ROLE.md/SKILL.md を読ませないと → ロールの境界が守られない → 越境が発生する
- 独立した作業を逐次実行すると → 不要な待ち時間が発生する → 並列起動で効率化する

### 方法論（手順）

pm-business は指示者として、タスクに必要な作業者（エージェント）を割り当てる。
エージェントにタスクを割り当てるのではなく、**タスクにエージェントを割り当てる**。

1. タスク分解書に基づき、タスクに必要なロールを特定する
2. タスクに各ロールのエージェントを Agent ツールで割り当てる
   - エージェントのプロンプトに以下を含める:
     a. 「まず `roles/<部門>/<ロール名>/ROLE.md` と `SKILL.md` を読め」
     b. タスク分解書の該当作業項目
     c. 受入基準のうち、そのロールに関係する項目
3. 依存関係のないエージェントは並列で起動する
4. 依存関係のあるエージェントは順次起動する（例: architecture → implementation）
5. 全エージェント完了後、タスクに review-gate エージェントを割り当てる

### 割り当てパターン

| 規模 | タスクに割り当てるエージェント |
|---|---|
| **Small** | implementation → review-gate |
| **Medium** | implementation + specialist（並列可） → review-gate |
| **Large** | architecture → implementation + specialist（並列可） → review-gate |

### 割り当て時のプロンプトテンプレート

```
あなたは {ロール名} ロール（作業者）として、以下のタスクに割り当てられました。
まず roles/{部門}/{ロール名}/ROLE.md と SKILL.md を読んでください。

## タスク
{タスク分解書の該当項目}

## 受入基準（あなたの担当分）
{受入基準の該当項目}

## 制約
- ROLE.md の Scope 内でのみ作業すること
- Boundary に該当する作業は行わないこと
- 完了時に成果物を報告すること
```

## SKILL-4: 用語統一チェック

### 論理構造（なぜこの手順か）

- 推定法の文脈で「粗利」を使うと → ユーザーが在庫法と推定法を混同する → 誤った経営判断につながる
- 色使いが統一されないと → 視覚的な手がかりが失われる → 実績と推定の区別がつかない

### 方法論（チェックリスト）

- [ ] 推定法の文脈で「粗利」を使っていないか（→「推定マージン」を使う）
- [ ] 在庫法のカードが緑系アクセント（sc.positive）を使っているか
- [ ] 推定法のカードがオレンジ系アクセント（palette.warningDark）を使っているか
- [ ] KpiCard に badge prop（actual / estimated）が設定されているか
