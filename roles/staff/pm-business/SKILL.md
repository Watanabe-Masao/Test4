# pm-business — スキル（論理構造 + 方法論）

## SKILL-1: タスク分解

### 論理構造（なぜこの手順か）

- ロール別に分解しないと → 責務が曖昧になる → 越境が発生し品質が低下する
- 受入基準が測定不能だと → 完了の判定ができない → タスクが永遠に「進行中」になる
- 作業順序を決めないと → 依存関係のあるタスクが並列で走る → 手戻りが発生する

### 方法論（手順）

1. タスクの内容を理解する
2. CLAUDE.md §ルーティング表 で主ロールと連携先を特定する
3. タスク規模を判定する（Small / Medium / Large）
4. 受入基準を定義する（測定可能な条件にする）
5. 作業項目と順序を決定する
6. タスク分解書を作成する
7. タスクに必要な作業者を決定する（SKILL-3 参照）

### 出力テンプレート（Medium / Large）

```
## タスク分解書

### タスク概要
（userからの依頼内容）

### タスク規模
Medium / Large

### 受入基準
- [ ] （測定可能な条件1）
- [ ] （測定可能な条件2）
- [ ] CI 6段階ゲート通過

### 作業者と順序
1. [architecture] （設計判断の内容）← Large の場合
2. [implementation] （実装内容）
3. [specialist/*] （専門検証の内容）← 該当する場合
4. [review-gate] 禁止事項チェック + CI 確認
5. [documentation-steward] ドキュメント更新 ← 必要な場合
```

### 出力テンプレート（Small）

Small タスクではタスク分解書を簡略化する。
1行の概要 + 受入基準のみで十分。作業分解は不要。

```
## タスク: （1行の概要）
- 規模: Small
- 受入基準: CI 通過 + （具体条件）
- 作業者: implementation → review-gate
```

## SKILL-2: 完了判定

### 論理構造（なぜこの手順か）

- review-gate が PASS でも受入基準を満たさない場合がある → 品質は通ったが要件は未達
- 受入基準の照合を省くと → 「動くけど要件と違う」コードが完了扱いになる
- 不足がある場合に追加指示しないと → 不完全な状態で次のタスクに進む → 技術的負債が蓄積する

### 方法論（手順）

1. review-gate の結果（PASS / FAIL）を確認する
2. PASS の場合、受入基準の各項目が満たされているか確認する
3. 全項目 OK → 完了を pm-business からuser に報告
4. 不足がある場合 → implementation に追加作業を連携指示する

## SKILL-3: タスクへの作業者決定

### 論理構造（なぜこの手順か）

- pm-business をスキップすると → タスク分解・受入基準が不在のまま作業が始まる → 完了判定できない
- タスクに必要な作業者を決めないと → 誰が何をやるか不明 → 責務の越境・重複が発生する
- 依存関係を考慮しないと → 設計が未決のまま実装が始まる → 手戻りが発生する

### 方法論（判断基準）

タスク分解書に基づき、以下の順序で必要な作業者を決定する。

1. ファイルパスベースルーティング（CLAUDE.md §ファイルパスベース自動ルーティング）で主ロールを特定
2. CLAUDE.md §ルーティング表 で連携先を特定
3. 規模に応じた作業者の組み合わせを決定:

| 規模 | 必要な作業者 |
|---|---|
| **Small** | implementation → review-gate |
| **Medium** | implementation + specialist → review-gate |
| **Large** | architecture → implementation + specialist → review-gate |

4. 依存関係を判断: architecture の出力が implementation の入力になるなら順次。独立なら並行可

## SKILL-4: 用語統一チェック

### 論理構造（なぜこの手順か）

- 推定法の文脈で「粗利」を使うと → ユーザーが在庫法と推定法を混同する → 誤った経営判断につながる
- 色使いが統一されないと → 視覚的な手がかりが失われる → 実績と推定の区別がつかない

### 方法論（チェックリスト）

- [ ] 推定法の文脈で「粗利」を使っていないか（→「推定マージン」を使う）
- [ ] 在庫法のカードが緑系アクセント（sc.positive）を使っているか
- [ ] 推定法のカードがオレンジ系アクセント（palette.warningDark）を使っているか
- [ ] KpiCard に badge prop（actual / estimated）が設定されているか
